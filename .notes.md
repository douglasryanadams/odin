# Notes

This is my project for learning how to use Claude to code as the primary author of a codebase. This was inspired after watching these YouTube videos from prominent software philosophers who all expressed enthusiasm for these new tools and the opportunities they present.

- [Martin Fowler](https://www.youtube.com/watch?v=CQmI4XKTa0U)
- [DHH](https://www.youtube.com/watch?v=JiWgKRgdgpI)
- [Kent Beck](https://www.youtube.com/watch?v=aSXaxOdVtAQ)

This file captures my notes for myself and future conversations about what I learned.

Firstly, Claude and agentic coding have evolved a ton in the last year. I read through a good bit of the Claude Code documentation and several new concepts jumped out at me since the last time I looked into this domain seriously.

1. The workflow and concepts are much more solidified than before, there's clear guidance about concepts and skills to develop to take advantage of Claude (and agentic coding tools generally). Explanations for first principles and primary concerns as a user of the tool are also laid out plainly.
2. Keeping track of the context window's capacity is a first principle skill to avoid exhausting tokens.
3. Common problems are clearly enumerated along with recommended solutions (using Claude.md, Agents, and Skills)
4. The CLI seems like the primary interface now, which I suppose makes sense because editing the code itself is less emphasized in this workflow. I'm now pair programming with a developer (Claude) that's writing the code, rather than writing the code myself and expecting it to help me.
5. A TDD mindset is helpful here, as a key recommendation is giving Claude tools to be able to measure its own success as it's iterating. Telling it how to tell when it's been successful seems critical.

For simple tasks that I know how to achieve traditionally, I just do those. Some of these:

1. `git init`
2. Adding to Github

As I'm working I'm realizing that just like a coworker, Claude is doing what it thinks the most average person would want but it can't intuit my own instincts and experience, I have to share those instructions and my philosophy explicitly. This presents an interesting challenge, I now need to be able to articulate by own style in a way that was implicit in the past.

I'm also finding it hard to keep my prompts focused and concise, instinctually I want to just rattle off all of the ideas I have. I almost need a separate file to keep track of the things that occur to me as I build the project out.

I also learned that to get Claude to respect my updates to CLAUDE.md, I need to restart the CLI (according to Gemini)

Keeping an eye on what it's doing with `watch tree` is something I should have started earlier

- Claude is somewhat "slow" it takes several minutes of processing and I'm seeing my usage climb in the UI.
- Speaking of usage, `/context` will visualize my context and help me keep track of my session
- Keeping the prompts concise and limiting the scope of the changes also makes it easier for me to review what it's doing iteratively

- After about 2 hours of iterating on this new project, I ran into my 5 hour rolling limit. To resume: `claude --resume 5304c536-e693-48bd-a399-76c2a669ba48`
    - I added $10 of extra usage, and I'm finding that each prompt costs me about 50 cents which really puts the cost of tools like this in perspective. I suspect this $10 will buy me an extra hour or so of pairing with Claude

- Clearing context between each increment of work (roughly a single commit) will keep things more efficient financially and in time.
- Reviewing the changes at each step with a critical eye can save a lot of wasted time. Some errors it makes are obvious if you pay attention, letting it figure itself out is expensive.


I accidentally updated something in my git status before doing a large task and it really slowed down/confused Claude. It spent a lot of time trying to figure out if it introduced the issue it was having or if it was something I did and started juggling git stash records to keep track of what was going on. Not only did this slow down Claude significantly and spend a lot more tokens, it also poluted my git environment making it harder for me, as the developer to keep track of what state the repo was in.

It's really really important to make sure that each step is its own commit to keep the commits reasonable. Each commit takes about 15-20 minutes so they stay concise and clear. I almost think that adding an instruction for Claude to create a commit after each task may be worthwhile.

There is an incredibly large amount of slop tutorials on YouTube that emphasize a lot of unnecessary features because they need to create content. And while the information is not innaccurate it generates so much noise that it's hard to find focused valuable content.

These are some of the prompts I provided, just so I can remember what I used and as examples

> Please set up a new Python project using uv for package management, and fastapi for the webserver
  using a standard gunicorn config and a docker file for running the web server. No need to include
  examples, do the bare minimum to demonstrate the project is set up and launches successfully


> Now let's add more helpful tools to the project, create a docker-compose file for running our stack
  using the docker image so that it reloads dynamically as we iterate on the codebase


> Rather than using the same docker image for dev and production, split it up into separate images,
  have the production image as the baseline and add the development and linting dependencies on top of
  it for the development and linting actions


> now lets set up testing, first lets set up unit testing for our python application using pytest, add
  a tests directory for this, and a preliminary test. In addition to the basic pytest configuration I
  would like to continuously measure code coverage, and test execution time.


> I would also like a way to keep an eye on complexity of the codebase as it evolves, please add tools
  that measure lines of code and cognitive complexity to run as part of our linting


> Review our current configuration for our linters and ensure that linting will fail if we exceed
  strict thresholds for each of them


> Now I want to plan a new feature, this feature will have have several capabilities and require
> integration with Claude's API for generative AI content. We will need to craft re-usable prompts as
> part of the solution so that we get consistently formatted responses from Claude's API. Please
> suggest improvements to this design as we work through the plan.
>
> \# Outcome
>
> When I search for a subject the subject is categorized into a person, place, event, or other
> category. For people, places, and events, I want a summary of those topics that highlights good
> things about them, bad things about them, a timeline of relevant events (like a biography for a
> person), and other important findings. I also want a general summary that helps orient me to the
> subject. For things that are not a "person" "place" or "event" we can return an error.
>
> \# Design
>
> I suspect this will be the general approach:
>
> 1. Send the search term to Claude's API for categorization (unless this is something searxng can do)
> into person place or event
> 2. Generate a series of well structured search phrase for searxng with Claude's API that will allow
> us to collect all the information we need
> 3. A parsing of search results to select 3-4 results worth investigating created by Claude's API
> 4. Our Python application will perform the search results, then use either a library like puppeteer
> to drive a browser to read the contents or rely on Claude's API to read the contents (not sure if
> Claude can do this)
> 5. Once all the content is retrieved, send back to Claude to summarize the contents, create the
> chronology of events, and other goals in our Outcome
> 6. Present these findings back to the UI, replacing the raw searxng response we currently display

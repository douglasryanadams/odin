# TODO

Tracking outstanding work that we want to address but haven't scheduled yet. Items are ordered by priority and magnitude of impact within each tier; revisit the ordering whenever a new TODO is added or scope changes.

## High priority

Strategic, high-impact work. Currently one large multi-phase initiative: taking direct ownership of our search backend. Ship each phase independently.

1. **Replace SearXNG with custom, first-party research clients.** SearXNG is causing more problems than it's worth: it's a separate meta-search container we don't control, only one of its engines (`braveapi`) actually returns results from our cloud IPs (its Mojeek, Startpage, Qwant, Karmasearch, and Wikipedia engines are all blocked or rate-limited), and it sits between us and the sources without giving us the control we need. Custom research is meant to be a core feature of this site, so we want to own the search layer directly. Replace it with our own clients: (a) a Brave client calling the Brave Search API directly, (b) a Wikipedia client using OAuth, and (c) Playwright-driven browser searches of Google, Bing, Yahoo, and any other engine that proves reliable.

   Approach: strangler-fig. Introduce a backend abstraction first, wrap the existing SearXNG call behind it so nothing breaks, add the new clients one at a time, then delete SearXNG once the new sources cover its role. Keep each phase shippable.

   - **Phase 1 — Search backend abstraction.**
     1. Define a `SearchBackend` protocol (`async search(query) -> list[SearchResult]`) and lift the `SearchResult` model out of `src/odin/searxng.py` into a neutral module (e.g. `src/odin/search/models.py`). Write and confirm the contract test before implementing.
     2. Add an aggregator that fans a query out to all enabled backends concurrently, dedupes by URL, merges the `engines` field, and degrades gracefully when a backend errors or times out (per-backend timeout, partial results allowed).
     3. Wrap the current SearXNG call as the first `SearchBackend` implementation and route `pipeline.py:_throttled_search` through the aggregator. No behavior change yet; this is just the seam.
     4. Add per-backend enable flags and timeouts to `src/odin/config.py`.
   - **Phase 2 — Brave client (direct Brave Search API).**
     1. The `BRAVE_API_KEY` is already provisioned and flows Secrets Manager → `.env` → compose into the SearXNG container. Re-plumb it straight into the web service config instead, so the key is no longer substituted through the SearXNG entrypoint template.
     2. Write a Brave client that calls the API directly and maps the response to `SearchResult`. Confirm tests against recorded fixtures before implementing.
     3. Register it as a backend and verify it matches or beats the current SearXNG + `braveapi` results.
   - **Phase 3 — Wikipedia client (OAuth).**
     1. Generate a Personal API Token on `meta.wikimedia.org` (auto-approved, server-side bot use case) and store it as `WIKIMEDIA_API_TOKEN` via the same Secrets Manager → `.env` → compose passthrough used for `BRAVE_API_KEY`.
     2. Write a Wikipedia client that calls `api.wikimedia.org/core/v1/wikipedia/en/search/page` with an `Authorization: Bearer ${WIKIMEDIA_API_TOKEN}` header (this auth path is what SearXNG's bundled `wikipedia` engine could not do, which is why it 429s and is disabled), and maps results to `SearchResult`. Confirm fixture-based tests first.
     3. Register it as a backend. (Brave already surfaces some Wikipedia URLs, so capture before/after coverage to confirm the dedicated client adds value.)
   - **Phase 4 — Browser SERP backends via Playwright (Google, Bing, Yahoo).**
     1. Spike first: determine which engines return usable organic results from our cloud IPs and survive bot walls. SearXNG's scraping engines are already blocked from these IPs, so expect the same risk and pick the most reliable strategy before building. If browser-driven scraping proves too unreliable, fall back to a paid SERP API for Google coverage (e.g. Serper, Developer plan ~$50/yr for 50k queries, 2,500 free/mo). This supersedes the former standalone "add Serper as a SearXNG `json_engine`" TODO.
     2. Build SERP backends that reuse the shared Playwright browser and anti-bot setup in `src/odin/fetch.py` (locale, `storage_state`, TLS impersonation), navigate to the chosen engine, and extract organic results into `SearchResult`. Test against saved SERP HTML, not live hits.
     3. Register them as best-effort backends and let the aggregator drop them silently when an engine serves a CAPTCHA.
   - **Phase 5 — Cut over and remove SearXNG.**
     1. Flip the aggregator to the new backends; delete the SearXNG backend wrapper, `src/odin/searxng.py`, `searxng_url` in `config.py`, and `get_searxng_url()` in `app.py`.
     2. Remove the `searxng` and `searxng-valkey` services from `compose/docker-compose.yml` and `compose/docker-compose.prod.yml`, and delete the `searxng/` directory (`entrypoint.sh`, `settings.yml.tmpl`).
     3. Drop or replace the SearXNG tests (`tests/integration/test_searxng.py`, `tests/test_searxng_settings.py`, the SearXNG assertions in `tests/test_compose_config.py`) with backend and aggregator tests.
     4. Update CI (`.github/workflows/ci.yml`): remove the `BRAVE_API_KEY=ci-dummy` shim that existed only to boot SearXNG, and supply whatever the new clients need instead.
     5. Update docs: replace `docs/searxng.md` with a `docs/search.md` describing the new architecture, and revise `docs/configuration.md`, `docs/backend.md`, `docs/aws-setup.md`, and the README "Where to go next" table.
   - **Phase 6 — Per-backend reliability and observability.**
     1. Add per-backend metrics (latency, error rate, result counts) and surface partial-result messaging to the user. This is the search-side half of "Harden search-result reliability" in the Medium tier; build it into the aggregator and close the loop with that item.

## Medium priority

Reliability work, UX bugs in normal flows, and provisioning that unblocks new search backends.

1. **Harden search-result reliability (remainder).** The first slice — swapping SearXNG's scraping `brave` engine for the official `braveapi`, pruning the consistently blocked `startpage` / `qwant` / `karmasearch` scrapers, and locking the API key inside a tmpfs — landed alongside the Brave API key provisioning. Outstanding: Python-side retries with backoff around the search call and the five Claude calls in `claude.py`, per-source timeouts in the pipeline, partial-result user-facing messaging (e.g., "profile built from 3 of 5 sources"), and per-stage failure metrics. Each is a separate small increment. Coordinate the search-side work with the backend rebuild (High priority #1): build the search retries, per-source timeouts, and failure metrics into the new aggregator (Phase 6) rather than against `searxng.search`, which that work removes.
2. **Dismissing the privacy policy banner mid-search restarts the search.** Clicking "Got it" on the privacy policy notice while a search is in progress causes the in-flight search to restart. The banner dismissal should be independent of the search lifecycle (no full page reload / form resubmit).
3. **Citations occasionally fail to render on results.** Some answer renders show no citations even though the answer text references the search results, and a retry on the same query usually brings them back. The sources are empirical — we read them off the search backend — so this is a wiring bug between the pipeline's source list and the template, not a Claude output issue. Investigate where the list gets dropped or filtered to empty (streaming event order, what the template was handed at render time, partial-result handling), fix the wiring, and add a regression test that pins the contract between pipeline output and template input. While in that code path, add a guardrail: if no readable sources reach the synthesis step, refuse to call Claude and show the user a clear "no usable sources for this query" message instead, so the model cannot fall back to its prior and fabricate an answer without empirical grounding.
4. **Generate an Open Graph share image.** The SEO baseline shipped text-only social unfurls. Add a 1200x630 PNG (`static/og-image.png`) with the ODIN wordmark on the existing dark background, reference it as `og:image` / `twitter:image` in the `social_meta` block of `_base.html`, and switch `twitter:card` to `summary_large_image`. Social platforms (LinkedIn, X, Slack, Discord) typically triple CTR with a real card image.
5. **Wire Google Search Console and Bing Webmaster verification meta tags.** Once `odinseye.info` is registered in both consoles, drive the verification tokens off optional env vars (`GOOGLE_SITE_VERIFICATION`, `BING_SITE_VERIFICATION`) so the values stay out of the repo, render them in `_base.html` only when present, and submit `sitemap.xml` from each console. Bing Webmaster can import the GSC verification directly.
6. **Delete account: friendly error when email does not match.** Submitting an email that is not the logged-in user's to the delete account form returns a raw JSON body in the browser instead of a formatted, useful error message. Render the error inline on the account page (or equivalent) the same way other validation errors are shown.
7. **Replace the removed `detect-secrets` lint step.** We pulled `detect-secrets` out of `make lint` because it shells out to `git` from inside the container, and a worktree's `.git` pointer escapes the bind mount, so the step fails any time someone lints from a worktree. We still want a pre-merge secret-scan signal — evaluate alternatives that don't depend on container-internal git: a GitHub Actions job using `trufflehog` or `gitleaks` over the PR diff, `git-secrets` pre-commit, or a host-side wrapper that runs detect-secrets outside Docker. Pick one, wire it into CI, document the audit workflow in `docs/configuration.md`.

## Low priority / backlog

Discretionary features and exploratory research; pick these up when higher tiers are clear or when one becomes strategically interesting.

1. **Display a representative photograph of the subject on the profile page.** Pick one image URL per profile from the same search results the pipeline already cites and render it in the sidebar above the Subject Compass, with a caption and the source domain framed in brackets (`[wikipedia.org]` style) to match the rest of the design. The photo must respect the same source provenance as the rest of the profile — never pull from an uncited source. Decide on image hosting (proxy through nginx with a short-lived cache vs link the source URL directly: direct is simpler but risks hotlinking complaints and leaks the viewer's IP to the source), fallback when no usable image is found (omit the slot, vs render a stylized `[ no portrait ]` placeholder), and how to handle sensitive subjects (minors, recent events with privacy implications, deceased individuals whose families have objected). Have Claude select the image during synthesis so the choice is reasoned-about rather than first-available.
2. **GitHub source link on the results page.** Restore a visible link to the project's GitHub repo so users browsing search results can find the source, read the README, or open issues — a trust signal and a discoverability path for contributors. Place it somewhere unobtrusive (results-page header corner or footer) — not the home page. Use the standard GitHub Octocat mark, styled to match the existing theme (current accent color, spacing, and corner radius) and small enough that it doesn't compete with the answer for attention.
3. **Recent successful searches on the home page.** Surface a list of recent user searches that succeeded on the home page, so visitors get a sense of what the tool can do and what others are finding. Decide on privacy/anonymization rules before implementing.
4. **Print/export button on search results.** Add a button on the results page that produces a clean, shareable version of the answer and sources (print-friendly stylesheet at minimum; ideally also a downloadable export such as PDF or Markdown). Strip site chrome, keep citations linkable, and make sure it works without JavaScript where feasible.
5. **Map of key event locations in the overview.** When the overview describes events tied to specific places, render a map alongside the summary that pins those locations. Have Claude emit structured place data (name + coordinates, or names we geocode server-side) so the map is driven by the answer rather than guessed from prose. Decide on a map provider (tile source, licensing, offline/static vs interactive) before implementing.
6. **Popularity / trends graph on results.** Show a time-series "popularity" chart for the search topic (Google Trends or an equivalent source) so users can see how interest in the subject has changed over time. Evaluate available APIs for licensing, rate limits, and historical depth; cache results to stay within quota and keep the page fast.
7. **Evaluate additional public-data search sources.** Investigate which public platforms expose supported APIs we could query to supplement our backends — Twitter/X, Reddit, Hacker News, Stack Exchange, Mastodon, YouTube, GitHub, etc. For each, capture: API availability and stability, auth requirements, rate limits, pricing, terms-of-service constraints on resale/redistribution, and how well the content fits our answer-synthesis pipeline. Output a short ranked recommendation of which to integrate first. (Brave and Wikipedia are out of scope here — they're committed in High priority #1.)
8. **IndexNow ping on publish.** Add a tiny background task that posts canonical URLs to the IndexNow endpoint whenever a sitemap-listed page changes (or on deploy). Near-zero effort to gain Bing and Yandex near-instant indexing; Google has not adopted IndexNow so this complements, rather than replaces, sitemap submission.

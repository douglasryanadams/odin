# TODO

Tracking outstanding work that we want to address but haven't scheduled yet. Items are ordered by priority and magnitude of impact within each tier; revisit the ordering whenever a new TODO is added or scope changes.

The current product direction is an opt-in deep research mode, layered on top of the existing fast profile. The fast 30 second profile stays the default front door; deep research is a bounded, agentic mode that runs more rounds of search, looks for connections across sources, and narrates its reasoning as it works. The initiative is sequenced below as slices 1 through 4: slice 1 (bounded iterative search) enters in High once the grounding and reliability fixes ahead of it are done, and slices 2 through 4 follow in Medium. Reliable citations and graceful degradation are hard prerequisites, because the cross-source connection step is the highest fabrication risk in the product. Two constraints govern the initiative: every surfaced connection must be grounded in cited sources, and per query cost must stay bounded so the project stays financially sustainable as usage grows.

## High priority

1. **Citations occasionally fail to render, and synthesis can run with no usable sources.** Some answer renders show no citations even though the answer text references the search results, and a retry on the same query usually brings them back. The sources are empirical, read off the search backend, so this is a wiring bug between the pipeline's source list and the template, not a Claude output issue. Investigate where the list gets dropped or filtered to empty (streaming event order, what the template was handed at render time, partial-result handling), fix the wiring, and add a regression test that pins the contract between pipeline output and template input. While in that code path, add a guardrail: if no readable sources reach the synthesis step, refuse to call Claude and show the user a clear "no usable sources for this query" message instead, so the model cannot fall back to its prior and fabricate an answer without empirical grounding. This is the grounding prerequisite for the deep research initiative: a mode that surfaces cross-source connections cannot ship on a substrate that sometimes drops citations or fabricates.
2. **Python-side retries with backoff around Claude calls.** Each of the five Claude calls in `claude.py` (categorize, generate_queries, select_urls, synthesize, assess) is a single attempt today: a transient 5xx or rate-limit fails the whole pipeline and burns the user's daily quota slot. Add bounded retries with exponential backoff on the SDK's retryable errors, plus per-stage failure metrics. This matters more under deep mode, which multiplies the number of sequential calls per query.
3. **Surface partial-result messaging to the user.** Per-backend timeout and graceful degradation already shipped with the aggregator (one backend down does not sink the query; partial results allowed). The outstanding user-facing half is telling the user when a profile was built from a subset of sources (for example "profile built from 2 of 3 sources"), so the degradation is honest rather than silent. Build it where the pipeline reports source counts to the template. The per-backend metrics half of this work moved to Medium.
4. **Deep research (opt-in), slice 1: bounded iterative search.** Today the pipeline is a single shallow pass: generate 3-5 queries, search, select up to 5 URLs, synthesize once. Introduce an opt-in deep mode that, after an initial pass, lets the agent read what it found, identify gaps and threads, and issue one or two more targeted rounds of queries and fetches before final synthesis. Cap the number of extra rounds hard: bounded depth keeps both cost-to-serve and the maintainer's cognitive load in check, and leaves the fast default profile untouched. This is the spearhead of the direction; the Medium-tier slices build on the loop it establishes. Start it only after the grounding fix in High #1, so deeper search cannot amplify ungrounded output.

## Medium priority

Quick reach wins sit at the top of this tier for momentum; the substantive thrust is the deep research initiative (slices 2 through 4 here, continuing from slice 1 in High). Browser SERP backends (Google/Bing via Playwright, or a paid SERP API) remain deferred in the backlog; revisit only if Brave plus Wikipedia coverage falls short.

1. **Generate an Open Graph share image.** The SEO baseline shipped text-only social unfurls. Add a 1200x630 PNG (`static/og-image.png`) with the ODIN wordmark on the existing dark background, reference it as `og:image` / `twitter:image` in the `social_meta` block of `_base.html`, and switch `twitter:card` to `summary_large_image`. Social platforms (LinkedIn, X, Slack, Discord) typically triple CTR with a real card image. This is the generic baseline; slice 4 later upgrades it to a per-profile card featuring the subject's photo, which converts far better. A roughly one-hour task, slotted first for momentum.
2. **Deep research (opt-in), slice 2: cross-source connection pass.** A synthesis step that explicitly looks for corroboration, contradiction, and links across the gathered sources rather than summarizing each in isolation. This is where the product's distinctive value lives: surfacing connections not readily apparent to a human reader. It is also the highest fabrication risk in the product, so every asserted connection must cite the specific sources it bridges; a connection without its supporting citations does not ship. Depends on slice 1 and on the High priority grounding fix.
3. **Deep research (opt-in), slice 3: narrate the agents' reasoning.** The pipeline already streams stage events to the page; upgrade the terse stage labels into a readable narrative of what the agent is doing and why ("found a gap in the early-career sources, searching for ..."). This turns the wait into the product, signals the research-automation positioning, and doubles as trust transparency by showing how the profile was built. Most of the work is prompt and frontend on the existing SSE infrastructure. Sequenced after slices 1 and 2, because the narrative is only compelling once there is real iterative research to narrate.
4. **Deep research (opt-in), slice 4: visual payoff, locations map and subject photo.** Once the deep pipeline emits structured data (places with coordinates, an image URL chosen from cited sources), render two high-reach surfaces. A map of the subject's key locations: for a person, birthplace, places of major accomplishment, and where they lived; for a place or event, the geography of what happened. Have the agent emit structured place data so the map is driven by the answer rather than guessed from prose, and respect source provenance. A representative photograph in the sidebar, chosen by the agent during synthesis from the same cited sources, captioned with the source domain in brackets (`[wikipedia.org]` style) to match the design; this image also feeds the per-profile Open Graph card. Both drive word-of-mouth and shareability, which is reach and not just on-page engagement; they sit last in the initiative because their payoff depends on the richer structured data the earlier slices produce. Decide before building: image hosting (proxy-and-cache versus direct link, weighing hosting cost and viewer-IP leakage against hotlinking complaints), the no-image fallback (omit the slot versus a stylized `[ no portrait ]` placeholder), map provider and tile licensing, and handling of sensitive subjects (minors, the recently deceased, families who have objected).
5. **Per-query cost-to-serve control.** Each query already costs five Claude calls (two on Sonnet) plus 3-5 Brave search calls, and `assess` re-sends the full fetched page content that `synthesize` already paid for. Trim what `assess` receives, or fold assessment into the synthesis call, and cache where inputs repeat. Low cognitive load, and it lowers the floor cost before deep mode multiplies per-query spend. As usage grows this is the lever that keeps the project sustainable and pushes back the point at which a paid tier or values-aligned advertising becomes necessary.
6. **Wire Google Search Console and Bing Webmaster verification meta tags.** Once `odinseye.info` is registered in both consoles, drive the verification tokens off optional env vars (`GOOGLE_SITE_VERIFICATION`, `BING_SITE_VERIFICATION`) so the values stay out of the repo, render them in `_base.html` only when present, and submit `sitemap.xml` from each console. Bing Webmaster can import the GSC verification directly.
7. **Dismissing the privacy policy banner mid-search restarts the search.** Clicking "Got it" on the privacy policy notice while a search is in progress causes the in-flight search to restart. The banner dismissal should be independent of the search lifecycle (no full page reload / form resubmit).
8. **Per-backend metrics and observability for the search aggregator.** Latency, error rate, and result counts per backend, so degradation is measurable rather than only logged; build it into `src/odin/search/aggregator.py`. Genuine operations value, but lower near-term user payoff at current traffic and real cognitive load to build and own, so it sits below the user-facing transparency work promoted to High.
9. **Replace the removed `detect-secrets` lint step.** We pulled `detect-secrets` out of `make lint` because it shells out to `git` from inside the container, and a worktree's `.git` pointer escapes the bind mount, so the step fails any time someone lints from a worktree. We still want a pre-merge secret-scan signal; evaluate alternatives that don't depend on container-internal git: a GitHub Actions job using `trufflehog` or `gitleaks` over the PR diff, `git-secrets` pre-commit, or a host-side wrapper that runs detect-secrets outside Docker. Pick one, wire it into CI, document the audit workflow in `docs/configuration.md`.
10. **Delete account: friendly error when email does not match.** Submitting an email that is not the logged-in user's to the delete account form returns a raw JSON body in the browser instead of a formatted, useful error message. Render the error inline on the account page (or equivalent) the same way other validation errors are shown.

## Low priority / backlog

Discretionary features and exploratory research; pick these up when higher tiers are clear or when one becomes strategically interesting.

1. **GitHub source link on the results page.** Restore a visible link to the project's GitHub repo so users browsing search results can find the source, read the README, or open issues, a trust signal and a discoverability path for contributors. Place it somewhere unobtrusive (results-page header corner or footer), not the home page. Use the standard GitHub Octocat mark, styled to match the existing theme (accent color, spacing, corner radius) and small enough that it doesn't compete with the answer. Near-zero cognitive load and zero cost to serve, so it leads this tier as a quick reach win.
2. **Recent successful searches on the home page.** Surface a list of recent user searches that succeeded on the home page, so visitors get a sense of what the tool can do and what others are finding. Decide on privacy/anonymization rules before implementing.
3. **Print/export button on search results.** Add a button on the results page that produces a clean, shareable version of the answer and sources (print-friendly stylesheet at minimum; ideally also a downloadable export such as PDF or Markdown). Strip site chrome, keep citations linkable, and make sure it works without JavaScript where feasible.
4. **Popularity / trends graph on results.** Show a time-series popularity chart for the search topic (Google Trends or an equivalent source) so users can see how interest in the subject has changed over time. Evaluate available APIs for licensing, rate limits, and historical depth; cache results to stay within quota and keep the page fast.
5. **Evaluate additional public-data search sources.** Investigate which public platforms expose supported APIs we could query to supplement our backends: Twitter/X, Reddit, Hacker News, Stack Exchange, Mastodon, YouTube, GitHub, and the like. For each, capture API availability and stability, auth requirements, rate limits, pricing, terms-of-service constraints on resale/redistribution, and how well the content fits our synthesis pipeline. Output a short ranked recommendation of which to integrate first. This gains importance under the deep research direction: more sources means more material for the cross-source connection pass to work with.
6. **IndexNow ping on publish.** Add a tiny background task that posts canonical URLs to the IndexNow endpoint whenever a sitemap-listed page changes (or on deploy). Near-zero effort to gain Bing and Yandex near-instant indexing; Google has not adopted IndexNow so this complements, rather than replaces, sitemap submission.
7. **Browser SERP backends (Google, Bing, Yahoo) or a paid SERP API.** Revisit only if Brave plus Wikipedia coverage falls short. Cloud-IP dependent and unverifiable from dev; expect the same bot-wall risk that has historically blocked third-party scraping from our IPs. Two paths: (a) Playwright SERP backends that reuse the shared browser and anti-bot setup in `src/odin/fetch.py` (locale, `storage_state`, TLS impersonation), navigate to the engine, and extract organic results into `SearchResult`, tested against saved SERP HTML; or (b) a paid SERP API for Google coverage (for example Serper, Developer plan around $50/yr for 50k queries, 2,500 free/mo). Either way it slots in as a `SearchBackend` via the registry in `src/odin/search/__init__.py`; register browser SERPs as best-effort and let the aggregator drop them silently on a CAPTCHA.
